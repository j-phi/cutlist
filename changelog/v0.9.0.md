# v0.9.0 — 2026-05-06

_Since [v0.8.0] (2026-05-06)._

A storage-shrinking release: doc image uploads now compress on the way in, scene thumbnails dropped to their intended size, and orphaned image assets get swept from IDB so projects don't bloat over time. The bundled demo project shrank from ~13 MB to ~1.4 MB as a result.

## Performance

- **Compress doc images on upload** — 4K phone photos were landing in IDB at multiple MB each. Every upload (toolbar, drop, paste) now routes through `browser-image-compression` in a worker, targeting 1920px / WebP q0.82 / 0.6 MB, and falls back to the original if re-encoding would inflate or the encoder fails. Adds a "Compressing N images…" indicator on the drop/paste path.
- **Scene thumbnails honor their pixel budget** — `captureThumbnail` was emitting 2048×1152 JPEGs (~280 KB) on Retina displays because `setSize` multiplies by the renderer's pixelRatio. Pinned pixelRatio to 1 around the capture, bumped the target to 1920×1080, and switched to WebP q0.8 — per-thumbnail bytes drop to ~110–180 KB.
- **Demo project re-exported** — `demo.cutlist` shrank from ~13 MB to ~1.4 MB after the compression changes above.

## Reliability

- **Sweep orphan build-doc image assets** — image uploads created via the editor used to linger in IDB after their `imageBlock` was deleted, after a mid-upload project switch stranded them, or after an import landed assets the doc didn't reference. `useBuildDoc`'s activeId watcher now fires a `deleteAssets` sweep over `stored − collectAssetIds(doc)`. Safe on fresh load because Tiptap history is in-memory only.
- **Lean `.cutlist.gz` exports** — `buildExportData` filters the same orphan set out at export time, so shared projects ship without dead asset payloads (without mutating IDB — the on-load sweep remains the source of truth).
