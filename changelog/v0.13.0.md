# v0.13.0 — 2026-05-12

_Since [v0.12.2] (2026-05-11)._

A dimension-accuracy release. CAD exporters that bake node rotation into vertex coordinates (Onshape, SketchUp Pro, Fusion 360) no longer cause rotated copies of a part to come out oversized or land in a different BOM group from their un-rotated twins. Cutlist now recovers the part's intrinsic dimensions from the geometry itself via an oriented bounding box, regardless of how the exporter chose to encode the part's pose.

## Bug Fixes

- **Rotated parts reported inflated dimensions** — GLTF and COLLADA imports computed each part's size from the world-aligned bounding box, so a 36×18×1 panel rotated 45° came out as ~38×38×1, and a vertex-baked-rotated twin of a flat panel landed in a separate BOM group. Sizes are now derived from an oriented bounding box over the real vertex positions, so rotated and un-rotated copies match exactly.

## Code Quality

- **New `obbDimsFromMeshes` helper** — Area-weighted face-normal clustering picks the heaviest cluster as axis 1, the heaviest perpendicular cluster as axis 2, and cross-products the third. O'Rourke's result (min-volume OBB aligns with a convex-hull face) means we avoid PCA's √2 worst case on square panels. Falls back to a caller-supplied AABB for degenerate geometry. Boxy parts only — faceted cylinders are a documented limitation.
- **GLTF JSON-walk derivation no longer inflates** — The synchronous `deriveFromGltf` path (used for fast tests and the legacy BOM-only path) now uses `local-AABB × matrix-scale` instead of the world AABB of transformed corners, so it no longer over-reports rotated nodes even without geometry data. The async builder still refines on top via OBB.
